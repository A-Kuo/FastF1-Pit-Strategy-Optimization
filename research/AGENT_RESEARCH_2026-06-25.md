## Engineering Insights (Production Patterns)

1. **Telemetry at Scale**: F1 telemetry infrastructure ingests 1.1M data points/second (1–1.5TB per race weekend) via WiMaX mesh network. Real-time processing (10–15ms latency) relies on Apache Kafka + Quarkus streaming. *Implication*: Your RL agent needs async telemetry ingestion; consider Kafka consumer for live data, Redis cache for feature vectors.

2. **FastF1 Caching & Pandas Integration**: FastF1 library already handles pit-stop data, tire strategy, and timing data as pandas DataFrames. Built-in caching prevents re-download overhead. *Implication*: Structure your data pipeline to use FastF1 as source-of-truth, cache preprocessed features in SQLite/Parquet for RL training.

3. **Python-First Architecture (Medium 2025)**: Winning teams use layered stacks—data ingestion → feature engineering → model (local inference or cloud) → orchestration → monitoring. RAG patterns and modular design enable fast iteration. *Implication*: Separate concerns: `src/telemetry_processor.py` (FastF1 wrapper), `src/strategy_agent.py` (RL core), `src/inference_server.py` (FastAPI for live decisions).

4. **State-Space Design**: Pit-stop strategy requires deep observation of tire degradation, fuel load, gap to competitors, safety car probability, and track conditions. Tire energy time-series prediction (XGBoost, as proven by Mercedes data) should feed RL observation. *Implication*: Build a feature extractor that outputs time-series windows of tire health; train RL agent on 50-lap rollouts.

5. **Interpretability for Adoption**: Research shows explainable RL (RSRL, arXiv:2501.04068) outperformed black-box strategies and earned stakeholder trust. Decision trees or attention mechanisms over pit-stop logic help justify recommendations. *Implication*: Add SHAP or attention visualization to strategy recommendations; report "pit-stop confidence" alongside decisions.
