## Summary
Pit-strategy ML has moved decisively toward RL and Bayesian decision frameworks since 2024, with several open-source RL simulators (PPO/DQN/DRQN) and at least four new arXiv papers (Jan–Dec 2025 window) formalizing race-strategy as a sequential decision problem under uncertainty. For this repo specifically, the highest-leverage findings are (1) the notebook-to-`src/`-package refactor pattern needed to fix the HTML/Jupyter-dominant language mismatch flagged in the last repo audit, and (2) the DRQN + Monte Carlo simulator design used by Mercedes-AMG PETRONAS's own published research, which is a natural reference architecture for a from-scratch strategy model.

## Engineering Insights
- **Cache FastF1 sessions locally.** `fastf1.Cache.enable_cache()` avoids re-hitting the F1 timing API on every run — this is the single biggest speedup mentioned across FastF1 tutorials and matters once telemetry pulls move into an automated pipeline rather than a manual notebook.
- **Adopt the `src/` layout to resolve the language-mismatch flag.** The repo audit noted GitHub reports HTML as the primary language because Jupyter notebooks dominate. The standard fix: keep exploratory notebooks in `notebooks/`, move all reusable logic into `src/<pkg>/` modules (data ingestion, feature engineering, model, evaluation), and have notebooks import from the package instead of containing raw logic. This is exactly the ask in `AGENT_TASK_QUEUE.md` priority #2.
- **Use `nbconvert` as the extraction starting point, not the final form.** Convert notebook cells to `.py` first, then manually refactor into functions/classes — projects that skip straight to copy-paste tend to retain notebook-style global state bugs.
- **Separate the RL environment from the policy.** Every RL repo surveyed (pit-stop-simulator, Race-Strategy-Simulator, TUMFTM) keeps a distinct `env/` (race/tire physics simulator) from `agent/` (PPO/DQN/DRQN policy) — this maps cleanly onto the requested `src/pit_strategy.py` + `src/telemetry_processor.py` split.
- **Benchmark against a non-ML baseline.** Both the DP/game-theory paper and the MINLP paper show classical optimization still competitive; an eval script that reports RL-agent-vs-classical-baseline gap (like the ~5s suboptimality figure) is more credible than RL-only metrics.

## Security Notes
- None of the surveyed repos expose API keys or credentials in source — FastF1 itself requires no auth (public F1 timing API), so there is no analogous credential-handling risk to import.
- Several RL simulator repos (pit-stop-simulator, Race-Strategy-Simulator) bundle Streamlit/FastAPI dashboards; if this repo adds a similar dashboard, apply the same rate-limiting guidance already flagged for the Yolov8 and CVE-KGRAG repos in `AGENT_TASK_QUEUE.md` before exposing any inference endpoint publicly.
- `fastf1.Cache` writes session data to local disk — ensure any cache directory is gitignored so multi-GB telemetry caches don't get committed.
